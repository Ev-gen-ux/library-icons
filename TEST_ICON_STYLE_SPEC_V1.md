# test-icon Style Spec v1 (20px)

Источник принципов (workflow): статья на Habr по дизайну иконок (форма, эстетическая целостность, узнаваемость; сетка; углы; кривые; оптимизация пикселей; толщина линий; общий “приём” внутри набора).  
Ссылка: [Поэтапное руководство по созданию иконок (Habr)](https://habr.com/ru/articles/304684/)

## Scope
- Применяется только к группе: `public/icons-20/test-icon/*`
- Размер: строго `20px`
- Цвет: монохром через `currentColor`, базовый цвет задавать как `color="#7D8394"` на корневом `svg`
- Стиль: современный SaaS, minimal/flat, высокая контрастность, чистая геометрия
- Никаких “размерных”/описательных слов в названиях файлов (например, `-small`, `-large`, `-muted`, `-filled` и т.п.). Разрешены только семантические имена (например, `user.svg`, `network-node.svg`).

## Output Requirements (SVG)
1. Каждый SVG должен быть:
   - с `width="20"` и `height="20"`,
   - с `viewBox` в едином формате для стабильности (рекомендуется `viewBox="0 0 24 24"`),
   - без встроенных “брендовых” цветов (кроме `#7D8394` как `color` на корне).
2. Все `fill`/`stroke` должны ссылаться на `currentColor`:
   - `fill="currentColor"` и/или `stroke="currentColor"`
   - запрещено использовать `stroke="#..."` и т.п. (кроме случаев, когда это прямо согласовано как исключение).
3. Фон и прямоугольники-контейнеры в SVG не рисовать (только прозрачные формы). “Чистый фон” обеспечивается страницей предпросмотра.

## Core Design Pillars (по Habr)
1. **Форма (Form):** силуэт и смысл собираются из простых геометрий (окружности/прямоугольники/дуги/сквиркл).
2. **Эстетическая целостность (Consistency):** в наборе повторяются одни и те же правила по углам, радиусам/терминалам, толщине и манере построения кривых.
3. **Узнаваемость (Recognizability):** каждая иконка объясняет себя 1-2 характерными элементами (например, для `user` — squircle-голова и минимальные плечи; для `network-node` — центральное ядро и симметричные “выходы”).

## Workflow (как разрабатывать иконки test-icon)
### 1. Начинайте с сетки
- Рабочая сетка: концептуально “как в статье”, но масштабированная под `viewBox 0 0 24 24`.
- Safe zone (свободная зона): оставляйте отступ от внешних границ viewBox.
  - Рекомендация: отступ ~`1.5` единицы от каждого края viewBox (аналог “2px свободной зоны” при масштабировании).
- Задача на этом шаге: положить ключевые объекты так, чтобы основная геометрия не “прилипала” к краям.

### 2. Первые эскизы — простые фигуры
- Всегда начинайте с грубого силуэта:
  - `user`: squircle head + shoulders/torso простой формы
  - `network-node`: центральное ядро (круг/скруглённый квадрат) + радиальные ветви (линейные сегменты + точки узлов)
- Не пытайтесь сразу “рисовать красиво” кривыми — сначала зафиксируйте форму.

### 3. Углы
- По возможности используйте углы, кратные **45°**.
- Если дизайн требует нестандартного угла — делите на секции с углами, кратными 45°, либо используйте кратные 15°.
- Технически это помогает получить более предсказуемое сглаживание на 20px.

### 4. Кривые
- Для сквиркла/скруглений:
  - используйте контролируемые геометрические построения (Bezier/shape tools), избегайте “ручной” кривой без контроля касательных.
- Симметрия:
  - ключевые элементы (лево/право, верх/низ) должны быть зеркально согласованы.

### 5. Оптимизация пикселей (crispness)
- Итоговые опорные точки (anchor points) и стыки линий старайтесь делать “сеточными”:
  - правило для стабильности: координаты опорных точек округлять так, чтобы они попадали на значения с шагом ~`0.5` (например, `x,y` в наборах вида `n`/`n+0.5`).
- Избегайте:
  - микросегментов и “почти вырожденных” дуг с очень маленькими радиусами,
  - асимметрии из-за разной точности десятичных чисел в зеркальных элементах.

### 6. Толщина линий / масса
Для test-icon допускается два режима:
- **Filled (предпочтительно):** силуэт задаётся заливкой; контуры формируются массой формы.
- **Stroke (допустимо):** только для связей/ветвей, где это улучшает читаемость.

Рекомендации для stroke-режима (если используете `stroke`):
- `stroke-width`: держать в диапазоне `1.2–1.5` для `viewBox 0 0 24 24`
- `stroke-linecap`: `round` (мягкие терминалы, как в требованиях)
- `stroke-linejoin`: `round` или `miter` в зависимости от угла, но не менять хаотично внутри одного набора иконок.

### 7. Единый “подпись”-приём внутри test-icon
Чтобы набор визуально отличался и оставался целостным, используйте единый приём для всей группы:
- “Node-SaaS”: повторяющаяся логика узлов и соединений или squircle-голова.
Практическое правило:
1) Для сетевых/узловых иконок — повторяйте одинаковый стиль узлов (одинаковый радиус/масштаб кругов) и одинаковую “геометрию” ветвей (линейные сегменты без декоративных изгибов).
2) Для профильных/человеческих — используйте squircle-голову с одинаковым характером скруглений (одинаковая степень “superellipse-like”).

### 8. Детали: умеренно
- На `20px` все детали должны быть читаемы без увеличения.
- Если элемент требует отдельного объяснения — он, скорее всего, лишний.
- Порог: избегайте деталей, сопоставимых по размеру с толщиной линии или меньше ~`0.8` единицы в viewBox.

## QA Checklist (обязательно перед добавлением иконки)
1. `width/height`: строго `20`
2. Размер в целом и по силуэту:
   - объект помещается в safe zone,
   - нет элементов, “вылезающих” за верхние/нижние/боковые границы,
   - симметрия соблюдена.
3. Геометрия:
   - углы чаще кратные 45°,
   - кривые построены контролируемо,
   - отсутствуют “мини-дуги” и чрезвычайно маленькие радиусы.
4. Криспость:
   - опорные точки не создают дрожание из-за дробных значений,
   - нет визуально “разваленных” углов после рендеринга.
5. Цвет и монохром:
   - все `fill/stroke` завязаны на `currentColor`,
   - на корне `svg` задан `color="#7D8394"`.
6. Стабильность с предпросмотром:
   - SVG валиден как XML,
   - есть `xmlns` (как минимум в существующих файлах).
7. Имя файла:
   - без размерных/описательных суффиксов.
