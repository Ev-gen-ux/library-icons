# icon-specs

Единая спецификация для иконок библиотеки в этом проекте: стиль, правила именования/организации, процесс добавления и группа `test-icon`.

## Источники принципов
- Правила построения (форма/целостность/узнаваемость, сетка, crispness, толщина/масса, умеренность деталей): [Habr](https://habr.com/ru/articles/304684/).

## 1) Общий стиль и требования (библиотека)

### 1.1 Цель
Собрать консистентную библиотеку полузалитых иконок под продукт:
- единый стиль,
- один цвет (управление через переменную/токен),
- совместимость с Inter,
- удобное масштабирование и стабильный процесс добавления новых иконок.

### 1.2 Обязательные правила (для библиотечных иконок)
- **Semi-filled (полузалитая) стилистика**: без визуального шума и микродеталей.
- **Цвет**:
  - продуктовые иконки — строго **монохром** (1 цвет через токен, практически: `currentColor`);
  - всё, что использует 3+ цветов, считается иллюстрацией, а не иконкой.
- **Размеры сетки**:
  - базовый ряд размеров привязан к шагу сетки (8-based): `16 / 24 / 32...`;
  - для продукта в этом проекте — обязательные размеры `24`, `20`, `16` (компоненты независимые, без Figma-variants).
- **Размер в этом репозитории**:
  - все SVG-иконки в кодовой базе хранятся и отдаются строго в размере **`20px`** (`width="20"`, `height="20"`); дополнительные размеры (`16/24/...`) ведутся только на стороне Figma.
- **Функциональная независимость размеров**: каждая иконка хранится как отдельный компонент для каждого размера.
- **Добавление новой иконки**: создается сразу в трех размерах `24/20/16`.

### 1.3 Структура в Figma (ориентир)
- Именование компонентов: `icon-{name}-24`, `icon-{name}-20`, `icon-{name}-16`.
- Группировка по категориям (внутри документа/библиотеки): navigation, actions, status, files, communication, commerce, system.
- Внутри документа — usage-правила: отступы, оптическое центрирование, поведение рядом с Inter.

## 2) Дизайн-стандарты (единый характер)

### 2.1 Геометрия и читаемость
- Единые принципы по форме: скругления, характер углов, пропорции пустот и заливок.
- Приоритет читаемости в малых размерах (особенно `16px`).
- Pixel-fitting + визуальная проверка в каждом размере.
- Оптический вес иконки должен быть согласован с Inter в типовых UI-решениях.

### 2.2 Цвет и монохромность
- Иконка перекрашивается **только через переменную/токен**.
- Для продуктовых иконок используется 1 цвет; маркетинговые/демо-иконки могут допускать до 2 цветов, но всё, что имеет 3+ цветов, относится к иллюстрациям.
- Строго избегаем смешения: одна иконка не должна сочетать в себе произвольные цветовые решения (например, однотонный glyph + разноцветные детали).

### 2.3 Принцип “форма → целостность → узнаваемость”
1) Сначала форма (силуэт) из простых геометрий.  
2) Затем эстетическая целостность (повторяющиеся правила углов/радиусов/терминалов/кривых).  
3) И только потом узнаваемость (1–2 характерных элемента на иконку).

## 3) Сетка, углы, кривые (Pixel + Optical grid)

Эти правила применяются ко всем новым иконкам (и особенно важны в small sizes).

### 3.1 Pixel grid (криспость прямых)
- “Якоря” (концы прямых отрезков, стыки, критичные точки кривых) задавайте так, чтобы они попадали в дискретные координаты.
- Практическое правило для стабильности в этом проекте:
  - координаты “округляйте” к значениям вида `n` и `n+0.5` в системе `viewBox 0 0 24 24`;
  - зеркальные элементы делайте симметрично по значениям, чтобы избежать дрожания из-за разной точности десятичных чисел.
- Векторный объект должен быть построен так, чтобы его центр масс совпадал с центром кадра 20×20 (иконка должна быть оптически центрирована как по вертикали, так и по горизонтали).

### 3.2 Optical grid (визуальный центр доминирующей формы)
- Центр масс доминирующей формы важнее математического центра контейнера.
- Для “оптически больших” фигур (круги/сквирклы) допускается микро-коррекция положения после первичного placement на pixel grid:
  - ориентир: 0.1–0.3 единицы по `x/y` (только при необходимости для визуальной центровки).

### 3.3 Edge padding (отступы от краёв)
- В Figma-референсе для фрейма 20×20 используется внутренний контур с обводкой `stroke = Inside`, поэтому:
  - красная обводка стоит по **внутреннему краю** фрейма;
  - расстояние от внешней границы фрейма до внутренней красной границы — **1 px со всех сторон**;
  - внешний контейнер: **20×20**, внутренняя “рабочая рамка” по красной обводке: примерно **18×18**.
- Для наших иконок принимаем эту же модель: векторный объект (силуэт или stroke) не должен выходить за внутреннюю 18×18 рамку:
  - для filled/силуэтных элементов внешняя граница доминирующего силуэта остаётся не ближе 1 px к краю фрейма;
  - если в иконке используются stroke-элементы (ветви/линии), крайние штрихи также держим не ближе 1 px к краю фрейма (расстояние между штрихом и краем не меньше ширины штриха).

### 3.4 Углы
- По возможности используйте углы, кратные **45°**.
- Если дизайн требует нестандартного угла:
  - делите на сегменты углов кратных 45° либо используйте кратные 15°.

### 3.5 Кривые и контролируемые формы
- Для сквирклов/скруглений используйте контролируемые геометрические построения (Bezier/shape tools), избегая “ручных” кривых без контроля касательных.
- Симметрия:
  - ключевые элементы (левая/правая часть, верх/низ) должны быть зеркально согласованы относительно центра кадра 20×20;
  - не должно быть случайных “перекосов”: опорные точки и радиусы для зеркальных частей должны совпадать по величинам (с точностью до шага 0.5 в координатах viewBox).

## 4) Толщина, масса, детали

### 4.1 Filled vs Stroke (общая логика)
- Предпочтение filled-логике: так выше узнаваемость, проще избежать “контурных” микродеталей.
- Если используете stroke:
  - **одинаковый stroke-width** по всему набору (для каждой “семьи” иконок);
  - расстояние между штрихами не должно быть меньше ширины штриха;
  - linecap/linejoin должны быть согласованы внутри набора.

### 4.2 Умеренность деталей
- На малых размерах детали должны читаться без “увеличения”.
- Избегайте деталей, размер которых сопоставим с толщиной линий или меньше ~0.8 единицы в viewBox.

### 4.3 Метафора и перспектива
- Сначала выбирайте **метафору**, а не “буквальный рисунок”: дом = “home”, секундомер = “timer”, лампочка = “idea” и т.п.
- При уменьшении иконки фокусируйтесь на сохранении метафоры, а не всех деталей крупной версии.
- Перспектива (3D/изометрия) используется либо как осознанный стиль всего набора, либо очень точечно, если это явно улучшает читаемость; по умолчанию — плоский (flat) вид.

## 5) QA-чеклист (общий)
- Иконка однозначно читается в каждом целевом размере (24/20/16).
- Не выбивается рядом с Inter в типовых UI-отступах.
- Корректно перекрашивается только через переменную/токен.
- Не нарушает стиль набора по форме, углам и визуальному весу.

### 5.1 Чистота вектора и Boolean-операции
- Для финальных иконок используйте Boolean-операции (Union/Subtract/Intersect/Exclude), чтобы сводить фигуры к минимальному числу слоёв.
- Не оставляйте внутри SVG “забытые” вспомогательные формы, дубликаты кривых и пересекающиеся контуры.
- Контуры и заливки должны быть приведены к чистой структуре, пригодной для оптимального экспорта SVG.

## 6) Именование и управление иконками

- **Имена по объекту, а не по смыслу**:
  - `stopwatch.svg`, а не `speed.svg`;
  - `lightbulb.svg`, а не `idea.svg`.
- Имена короткие, в `kebab-case`, без размерных суффиксов и визуальных характеристик (`-small`, `-big`, `-outline`, `-filled` и т.д.).
- Одна иконка — один master в библиотеке:
  - вариации (цвет/метка/валюта и т.п.) должны основываться на том же master через токены и описания, а не через дублирование SVG.

## 6) Стандартизированный промт для наполнения

```text
Сгенерируй иконку в стиле semi-filled для продуктовой библиотеки.

Контекст:
- Название: {icon_name}
- Категория: {category}
- Смысл/метафора: {meaning}
- Платформа: {web/mobile/both}

Обязательные требования:
- Полузалитая монохромная иконка, без градиентов и эффектов.
- Совместимость с Inter: визуальный вес и читаемость рядом с текстом.
- Сразу подготовить ТРИ независимых версии: 24px, 20px, 16px.
- Каждая версия должна быть самостоятельной (не масштаб-копия без оптической адаптации).
- Минимум мелких деталей, особенно для 16px.

Figma/структура:
- Без variants.
- Экспорт/наименование как отдельные компоненты:
  icon-{name}-24
  icon-{name}-20
  icon-{name}-16

Критерии приемки:
1) Все 3 размера читаются и визуально консистентны.
2) Цвет управляется одной переменной/токеном.
3) Иконка соответствует стилю библиотеки.
```

## 7) Дополнение: `test-icon` (локальная экспериментальная группа)

Группа `public/icons-20/test-icon/*` предназначена для быстрых тестов стиля/геометрии на одном размере `20px`.

### 7.1 Правила группы `test-icon`
- Размер: **только `20px`**.
- Цвет: монохром через `currentColor`; базовый цвет задавать на корневом `svg` как `color="#7D8394"`.
- Стиль: современный SaaS, minimal/flat, высокая контрастность, чистая геометрия.
- В названиях файлов запрещены размерные/описательные суффиксы (например, `-small/-large/-muted/-filled` и т.п.); допускаются семантические имена вроде `user.svg`, `network-node.svg`.

### 7.2 Требования к SVG в `test-icon`
- `width="20"`, `height="20"`.
- Единый формат `viewBox` (рекомендуется `viewBox="0 0 24 24"`).
- `fill`/`stroke` должны использовать `currentColor`:
  - `fill="currentColor"` и/или `stroke="currentColor"`.
- Запрещено рисовать “фон” внутри SVG (прямоугольники-контейнеры): фон обеспечивает страница предпросмотра.

### 7.3 QA для `test-icon`
- SVG валиден как XML.
- На корне `svg` задан `color="#7D8394"`.
- Нет “микро-дуг”/микросегментов с чрезвычайно маленькими радиусами.
- Криспость: якоря/точки не должны создавать дрожание при рендере.
